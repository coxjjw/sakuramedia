# 测试策略

本项目的测试目标是保护业务行为与跨层契约，而不是复刻页面实现。新增测试应优先放在能够以最低成本证明行为正确的层级。

## 分层原则

### 必须重点覆盖

- 网络与会话：鉴权、刷新令牌、错误映射、并发请求和持久化。
- API 与 DTO：请求方法、路径、参数、请求体、响应映射和后端错误保真。
- Controller / Notifier / Provider：加载、重试、分页、并发、失败恢复、销毁后的异步完成和状态广播。
- 路由：鉴权重定向、深链、返回路径、平台路由映射，以及真实页面能够接入应用组合根。
- 共享组件：对外回调、禁用状态、键盘和鼠标交互、可访问语义，以及容易回归的跨平台差异。

### 页面测试仅在必要时添加

页面级 widget test 只用于无法在更低层验证的关键接线或完整用户流程，例如：

- 表单提交确实把用户输入交给正确的状态对象；
- 关键操作打开确认流程，并把结果反馈到当前页面；
- 平台专属交互依赖真实 Widget 树才能验证；
- 曾发生过且不能由 Controller、共享组件或路由测试覆盖的回归。

同一加载、失败、重试、分页或状态转换已经由 Controller 测试覆盖时，Desktop、Mobile 和路由测试不再重复完整覆盖。路由测试负责“页面接得上”，页面测试负责少数“接线接得对”的场景。

## 断言边界

优先断言用户可观察结果和公开状态：

- 操作是否成功、失败是否可恢复；
- 请求是否正确、状态是否正确；
- 导航目标和返回行为是否正确；
- 关键内容、反馈与可访问语义是否出现。

通常不要断言：

- 私有 Widget 树层级或具体 Widget 类型；
- 普通文案、图标和装饰组件的精确数量；
- 页面内部的精确字号、间距、百分比高度和布局算法；
- 通过读取生产源码并搜索字符串来证明实现方式；
- 已由 Controller、API、共享组件或路由测试证明的同一分支。

设计系统的全局裸值约束集中保留在 `test/theme/theme_source_guard_test.dart`，不要在各业务组件测试中重复源码扫描。平台构建配置等无法通过 Dart 运行时观察的静态文件契约可以例外使用文件检查。

## 新增测试前检查

1. 这个失败会造成真实业务风险，还是只反映一次正常重构？
2. 能否用纯 Dart 单元测试验证，而不是 pump 整个页面？
3. 仓库里是否已有另一个层级覆盖相同行为？
4. 测试是否只依赖公开行为和稳定契约？
5. 如果 UI 结构变化但用户行为不变，这个测试是否仍应通过？

无法明确回答业务风险时，默认不新增测试。

## 运行方式

- 修改单个模块时，直接运行对应测试文件。
- 提交前运行 `./scripts/test.sh` 做全量验证。
- 修改生产代码后同时运行 `flutter analyze --no-fatal-infos --no-fatal-warnings`。

全量入口会动态发现 `test/` 下的测试并生成临时 bundle，因此删除或新增测试文件后不需要维护静态清单。
